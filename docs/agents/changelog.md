# 任务：生成中文更新日志并写入文件

你需要根据用户提供的“上一个 release 版本 commit ID”，获取该 commit 之后直到当前 `HEAD` 的所有 commit，分析这些 commit 的实际改动内容，并生成一段中文更新日志，向前追加到：

`docs/zh-cn/CHANGELOG.md`

只生成中文版本，不要生成其他语言版本。

新增功能只能出现在 `添加` 中；不要再把同一个新功能的文案、依赖、样式、设置项调整重复写进 `更新`。

## 输入

用户会提供：

- 上一个 release 版本的 commit ID：`<PREVIOUS_RELEASE_COMMIT_ID>`
- 当前版本号：`<VERSION>`，例如 `1.1.7` 或 `v1.1.7`

如果用户没有提供 commit ID，不要猜测，不要自行选择 tag，必须要求用户补充。

如果用户没有提供版本号，则按以下顺序尝试获取：

1. 从项目配置文件中读取版本号，例如：
    - `package.json`
    - `manifest.json`
    - `wxt.config.*`
    - 其他明显包含项目版本号的配置文件
2. 如果无法可靠确定版本号，必须要求用户补充，不要编造。

日期使用执行任务当天日期，格式为 `YYYY-MM-DD`。

## 获取 commit 范围

使用以下范围获取 commit：

```bash
git log --reverse <PREVIOUS_RELEASE_COMMIT_ID>..HEAD
```

要求：

- 必须获取该 commit 之后的所有 commit。
- 不要遗漏 merge commit，但总结时不要机械罗列每个 commit。
- 如果 commit message 信息不足，必须查看对应 diff。
- 需要结合 commit message、改动文件路径和代码 diff 判断真实更新内容。
- 不要只根据 commit message 瞎总结。

## 分析规则

请将所有改动按功能模块聚合，而不是按 commit 逐条列出。

模块名称要求：

- 使用简洁中文，例如：
    - `广告屏蔽`
    - `播放器界面`
    - `迷你播放器`
    - `音量增强器`
    - `评论翻译`
    - `i18n`
    - `依赖`
    - `构建`
    - `设置`
    - `存储`
    - `样式`

- 如果模块能从文件路径、功能名称或 commit scope 中明确判断，优先使用明确模块名。
- 如果只是依赖升级、构建配置、工程维护，统一归类到 `依赖`、`构建` 或 `工程维护`。
- 不要使用过于宽泛的模块名，例如 `其他`，除非确实无法判断。

## 分类规则

将更新内容分为以下三类：

### 更新

用于已有功能的改进、调整、优化、重构、翻译更新、依赖升级、性能优化等。

示例：

- 改进已有弹窗识别逻辑
- 优化存储写入策略
- 调整 UI 显示效果
- 更新翻译
- 升级依赖
- 重构已有模块

### 添加

用于新增功能、新增选项、新增规则、新增模块、新增配置能力等。

如果某个功能是本版本首次出现，那么它相关的界面文案、设置入口、默认行为、内部兼容、依赖支持、样式完善等内容，都应合并到 `添加` 中描述，不要再拆到 `更新`。

示例：

- 新增某个设置项
- 新增某个播放器功能
- 新增某类屏蔽规则
- 新增语言翻译
- 新增组件或模块

### 修复

用于 bug 修复、异常处理、兼容性修复、显示错误修复、逻辑错误修复等。

示例：

- 修复某功能失效
- 修复布局错位
- 修复配置不生效
- 修复特定场景下的异常
- 修复兼容性问题

## 写作要求

更新日志必须满足：

- 使用中文。
- 语言简洁、准确、像正式 changelog。
- 不要出现 commit ID。
- 不要出现开发过程描述。
- 不要出现“可能”“似乎”“应该”等不确定表达。
- 不要夸大功能。
- 不要编造 commit 中没有体现的内容。
- 不要把同一件事重复写到多个分类。
- 相近改动需要合并成一条。
- 每条内容必须能从 commit 或 diff 中找到依据。
- 如果某个分类没有内容，则不要输出该分类标题。

## 输出格式

严格使用以下格式：

```markdown
## [版本号] - YYYY-MM-DD

### 更新

- **模块名称:** 更新的功能描述。

### 添加

- **模块名称:** 添加的功能描述。

### 修复

- **模块名称:** 修复说明。
```

示例：

```markdown
## [1.1.7] - 2026-06-12

### 更新

- **广告屏蔽:** 新增 YouTube 购物推荐栏相关屏蔽规则。
- **禁用暂停:** 改进“视频已暂停，继续播放？”弹窗识别，避免误关其他弹窗。
- **记忆存储:** 将 500ms 内的重复写入合并处理，减少不必要的存储更新。
- **迷你播放器:** 简化弹出窗口中迷你播放器的尺寸显示。
- **i18n:** 更新新版播放器与回退设置相关翻译。
- **依赖:** 更新 WXT/构建工具，移除旧版浏览器全局变量兼容代码。

### 添加

- **迷你播放器:** 新增滚动迷你播放器，支持设置尺寸、位置、屏幕边距与触发距离。
- **音量增强器:** 新增真实音量增强功能，支持播放器内控制与自定义增强级别。
- **评论翻译:** 新增多行评论逐行显示译文的选项。
- **播放器界面:** 新增隐藏 YouTube 迷你播放器按钮的选项。
- **回退:** 新增回到旧版 YouTube 播放界面布局的选项。

### 修复

- **迷你播放器:** 改进进入/退出迷你播放器模式时的层级与布局恢复。
- **回退:** 修复开启旧版播放界面布局后，非 16:9 视频被塞进 16:9 外壳里显示的问题。
```

## 写入文件要求

将生成的更新日志追加到：

```text
docs/zh-cn/CHANGELOG.md
```

要求：

- 不要覆盖原文件内容。
- 不要删除已有历史记录。
- 按该文件现有 changelog 顺序插入新版本记录。
- 如果文件中已有相同版本号记录，不要重复追加；应更新已有版本段。
- 写入前先读取文件，判断现有格式和插入位置。
- 写入后检查文件内容，确认格式正确。

### 写入JSON格式化文件 `changelog.json`

读取当前项目的 `/changelog.json`，解析其格式，并将内容更新写入该文件。

## 最终回复

完成后只回复：

```markdown
已更新 `docs/zh-cn/CHANGELOG.md`。
```

如果无法完成，说明具体原因，例如：

```markdown
无法生成更新日志：未提供上一个 release 版本的 commit ID。
```

## 用户视角过滤规则

生成更新日志时，必须站在普通用户视角描述变化，而不是开发者视角。

更新日志只记录用户能感知到的功能变化、界面变化、行为变化、选项变化、体验优化和问题修复。

以下内容默认不要写入更新日志，除非它们直接影响用户可见功能：

- commit ID、分支、tag、文件路径、目录迁移。
- 内部数据结构、配置字段、布尔值、枚举值、兼容旧配置等实现细节。
- 依赖包名称、依赖安装、构建脚本、构建配置、版本号 bump。
- README、CHANGELOG、文档链接、版本源配置等纯维护改动。
- 重构、抽象、封装、类型调整、格式化、lint、测试、CI 等工程细节。
- “兼容旧版配置”“迁移旧数据”“保留旧字段”这类用户不需要知道的内部兼容逻辑。

如果某个内部改动确实会影响用户，应改写成用户能理解的结果，而不是暴露实现。

示例：

错误：

- **播放器按钮:** 将播放器按钮隐藏设置升级为“自动 / 隐藏 / 强制启用”显示方式，并兼容旧版布尔配置。

正确：

- **播放器按钮:** 优化播放器按钮显示设置，支持自动、隐藏和强制显示三种模式。

错误：

- **构建:** 更新版本到 1.1.8，调整全量构建脚本，并补充自定义 CSS 编辑器所需依赖。

正确：

- 如果只是构建脚本和依赖调整，不写入更新日志。
- 如果用户能感知到自定义 CSS 编辑器能力变化，则写成：
    - **自定义 CSS:** 完善自定义 CSS 编辑器相关支持。

错误：

- **文档:** 将中文 README 与更新日志迁移到 docs/zh-cn/，并更新 README、更新日志和版本源配置中的文档链接。

正确：

- 如果只是文档目录迁移，不写入更新日志。
- 如果文档内容确实补充了用户说明，则写成：
    - **文档:** 更新中文使用说明和更新日志入口。

错误：

- **i18n:** 更新倍速滑块、画面缩放、截图按钮、自定义 CSS、字幕翻译分组和配置记忆导入导出相关文案。

正确：

- **设置界面:** 优化倍速滑块、画面缩放、截图按钮、自定义 CSS、字幕翻译和配置导入导出相关说明文案。

错误：

- **自定义 CSS:** 新增自定义 CSS 编辑器。
- **自定义 CSS:** 补充自定义 CSS 编辑器依赖并优化相关文案。

正确：

- **自定义 CSS:** 新增自定义 CSS 编辑器，支持在设置中编写和管理自定义样式。

错误：

- **播放器按钮:** 新增播放器按钮显示模式。
- **播放器按钮:** 优化播放器按钮显示设置，支持自动、隐藏和强制显示三种模式。

正确：

- **播放器按钮:** 新增播放器按钮显示模式，支持自动、隐藏和强制显示三种选项。

错误：

- **字幕翻译:** 新增字幕翻译分组。
- **设置界面:** 更新字幕翻译分组相关文案。

正确：

- **字幕翻译:** 新增字幕翻译设置分组，优化相关选项说明。

## 信息降噪规则

对 commit 内容进行总结时，必须主动合并、降噪、改写。

要求：

- 不要逐条翻译 commit message。
- 不要把内部实现词汇原样搬进更新日志。
- 不要让用户看到“布尔配置”“依赖”“脚本”“路径迁移”“版本源配置”这类开发细节。
- 一条更新日志应该描述“用户看到什么变化”，而不是“代码怎么实现”。
- 如果某项改动没有用户可感知结果，直接忽略。
- 如果多个 commit 都是在完善同一个功能，只合并成一条。
- 如果某个功能既有内部兼容又有界面变化，只写界面变化。
- 如果不确定是否用户可感知，优先不写。

## 分类互斥规则

同一个用户可感知功能只能出现在一个分类中，禁止同时出现在 `添加` 和 `更新` 中。

优先级如下：

1. 如果某个功能、选项、模块、按钮、规则或能力是本版本新加入的，必须归类到 `添加`。
2. 新功能相关的文案补充、设置项调整、默认值调整、样式完善、内部兼容、依赖补充、后续小优化，默认合并进该功能的 `添加` 描述中，不要再单独写到 `更新`。
3. 只有已经在上一个版本存在的功能，在本版本被改进时，才写入 `更新`。
4. 如果某个修复只发生在本版本新加功能的开发过程中，且用户没有在旧版本中遇到这个问题，不要写入 `修复`，直接忽略或合并进 `添加`。
5. 如果某个功能既有新增能力，又有对旧功能的独立改进，需要拆开描述，不能把新旧内容混成一条。

判断标准：

- 旧版本没有这个功能：写 `添加`。
- 旧版本已有这个功能，本版本只是增强、调整、优化：写 `更新`。
- 旧版本已有这个问题，本版本解决了：写 `修复`。
- 新功能开发过程中顺手修的内部问题：不写。
